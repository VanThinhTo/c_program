Fix Notes

Scope: reviewed the C source exercises in this repository. The PDF files under C/ and C/Exercises/ look like course/reference material, so the actionable debug comments below focus on source code.

Checks used:

  • clang -std=c11 -Wall -Wextra -Wpedantic -fsyntax-only on each .c file.
  • Manual source review for runtime behavior, input validation, array bounds, and exercise requirements.

Highest priority fixes

  1. main.c does not compile through utils/import.h.
  • utils/import.h includes .c implementation files directly. This pulls in extra main functions and duplicate/conflicting function definitions.
  • utils/power_ex.c defines power(float, unsigned) and has its own main, while utils/pow.c defines power(double, unsigned).
  • utils/recursive.c defines two different functions named pattern_2.
  • Suggested fix: make utils/import.h contain only prototypes, remove all #include "file.c" lines, compile .c files separately, and give duplicate functions unique names.
  1. Several files compile only as standalone exercises, not together.
  • Many files define their own main, so they cannot be linked into one binary at the same time.
  • Suggested fix: either keep one active exercise main at a time, or move reusable functions into utility files and keep exercise drivers separate.
  1. Input validation is inconsistent.
  • Many exercises use scanf or fgets without checking the return value.
  • Many array exercises create VLAs from unchecked input, for example int arr[n].
  • Suggested fix: always validate scan success and require positive sizes before declaring arrays.
  1. Unbounded string input appears in several places.
  • Examples: scanf("%s", s) and scansets without widths.
  • Suggested fix: use field widths such as scanf("%99s", s) and %199[...].
  1. Some functions have undefined behavior from out-of-bounds array access.
  • Major examples: utils/array.c, pra/exam_10.c, and exam/example.c.
